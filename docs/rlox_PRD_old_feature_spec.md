# Reinforcement learning frameworks and the case for Rust

**Python dominates the RL framework landscape, but its fundamental architecture—the GIL, interpreter overhead, and memory unsafety from C extensions—creates performance ceilings that cost researchers 5–50× in throughput.** Reimplementing performance-critical RL bindings in Rust offers a proven path to reclaim this lost performance, following the same hybrid pattern that delivered Polars (3–10× over Pandas), Pydantic v2 (5–50× over v1), and Ruff (10–100× over flake8). The Rust RL ecosystem remains immature—no single library matches Stable-Baselines3's completeness—but the foundational pieces (burn at 11k GitHub stars, candle at 19k, PyO3 at 13k) are production-ready and accelerating. The optimal strategy is not a full rewrite but selective Rust reimplementation of environment simulation, replay buffers, and parallel stepping, exposed to Python via PyO3.

## The four Python RL frameworks each solve different problems

**Stable-Baselines3** (v2.7.1, ~12.5k GitHub stars) is the workhorse. Its scikit-learn-inspired API (`model = PPO("MlpPolicy", "CartPole-v1"); model.learn(10000)`) makes RL accessible, with 7 core algorithms (PPO, SAC, TD3, A2C, DQN, DDPG, HER) plus 7 more in SB3-Contrib (TRPO, RecurrentPPO, MaskablePPO, QR-DQN, TQC, CrossQ, ARS). It ships with VecEnv abstractions, rich callbacks, and Hugging Face model sharing. The trade-off is rigid: **single-machine only, single-agent only, no model-based RL, no offline RL**. Its SBX (Jax) variant reveals SB3's core bottleneck—SBX runs **up to 20× faster** than PyTorch SB3 on identical tasks, exposing how much performance the Python/eager-mode stack leaves on the table.

**CleanRL** (~9k stars) takes the opposite architectural bet. Every algorithm lives in a single, self-contained file—PPO on Atari is **340 lines of code**, not 15 files across a class hierarchy. This makes CleanRL exceptional for education and rapid prototyping but unusable as a library (you cannot `import cleanrl`). A 2025 study across 56 Atari games found SB3 and CleanRL **achieve statistically equivalent PPO performance**, with both reaching superhuman scores in 50% of trials. Meta's LeanRL fork of CleanRL demonstrates that adding `torch.compile` and CUDA graphs delivers **6.8× speedups for PPO** and **5.7× for SAC**—purely by eliminating Python interpreter overhead in the training loop.

**TorchRL** (v0.11.1, ~3.3k stars) from Meta represents the most architecturally ambitious approach. Built around TensorDict—a batched, device-aware tensor container—it keeps data on GPU and avoids the numpy-to-tensor conversions that plague other frameworks. The ICLR 2024 benchmarks are striking: TorchRL's async collector achieves **19,401 fps on Breakout** versus RLlib's 97 fps and **32,894 fps on HalfCheetah** versus RLlib's 1,868 fps. GAE computation takes **1.33ms** versus SB3's 14.1ms (10.6× faster). TorchRL covers online RL (PPO, SAC, DQN, IMPALA), offline RL (IQL, CQL, Decision Transformer), model-based (DreamerV3, TD-MPC), and multi-agent (MAPPO, QMIX via BenchMARL). Its beta status means API churn remains a concern.

**RLlib** (Ray 2.54.0, part of Ray's ~41.5k-star monorepo) is the only framework built for distributed training. It scales across three axes: `num_env_runners` for parallel data collection, `num_envs_per_env_runner` for vectorized sub-environments, and `num_learners` for data-parallel GPU training. This cluster-scale architecture comes at a cost: substantial Ray overhead from serialization, scheduling, and actor management. Users on GitHub consistently report difficulty replicating baseline algorithm performance, complex configuration, and slow convergence on simple environments compared to SB3 or CleanRL. The "new API stack" transition (from 8 user-facing classes to 5, now PyTorch-only) has improved design but added migration burden. RLlib's algorithm coverage on the new stack is narrower—PPO, DQN, SAC, APPO, IMPALA, DreamerV3—with many old-stack algorithms (DDPG, TD3, MADDPG, AlphaZero) not yet ported.

## Python's architectural ceiling limits every framework

The GIL is the single most damaging bottleneck. DeepMind's Manuel Kroiss stated publicly (in PEP 703 discussions) that even with fewer than 10 threads, the GIL becomes the bottleneck for their RL workloads—where they need 50–100 threads per process. This forces RL frameworks into multiprocessing with expensive IPC. SB3's SubprocVecEnv is actually **5× slower** than sequential DummyVecEnv for simple environments like CartPole because IPC overhead exceeds computation time. Even for complex environments, speedup plateaus well below linear scaling.

EnvPool (NeurIPS 2022) proved the magnitude of this gap by reimplementing environments in C++ with a thread pool: **1 million Atari frames/second** on 256 CPU cores—**14.9–19.6× faster** than Python subprocess-based vectorized environments. For MuJoCo, the gap widens to **19.2×**. The Gymnasium documentation itself acknowledges: "environments can be slow to run compared to custom C, C++, or Cython-based environments."

Training loop overhead compounds the problem. RL uses shallower networks than supervised learning, making Python interpreter overhead proportionally larger. LeanRL's benchmarks show that `torch.compile` alone—which eliminates Python↔C++ boundary crossings—delivers **2.7–6.8× speedups** across PPO, SAC, and TD3 without changing any algorithm logic. TorchRL's advantage over SB3 (10.6× faster GAE) comes largely from avoiding numpy-based operations and keeping data GPU-resident.

Memory safety creates a different class of problems. MuJoCo rendering segfaults in multithreaded contexts (documented in mujoco-py#283). Memory leaks accumulate during long training runs—Google DeepMind's own MuJoCo issue tracker documents leaks when reloading XML models with callbacks. RLlib's ReplayBuffer uses Python lists, meaning "if the required data does not fit into memory, training is not possible." These are not edge cases; they are structural consequences of mixing Python with C extensions across thread and process boundaries.

Reproducibility remains fundamentally unsolved. Henderson et al.'s "Deep Reinforcement Learning that Matters" (AAAI 2018) showed different random seeds produce completely different learning curves. Agarwal et al. (NeurIPS 2021 Outstanding Paper) found that **50–70% of claimed RL improvements may be spurious** when properly accounting for statistical uncertainty. PyTorch's own documentation warns that "completely reproducible results are not guaranteed across PyTorch releases, individual commits, or different platforms," driven by non-associative floating-point operations in CUDA's atomicAdd and non-deterministic cuDNN algorithm selection.

Multi-agent support is fragmented: RLlib offers the most mature MARL (managing **70k+ actions/s/core** at 10,000 agents), TorchRL integrates BenchMARL/VMAS for MAPPO and QMIX, CleanRL provides basic PettingZoo examples, and SB3 has no MARL support at all.

## Rust's RL ecosystem is immature but its foundations are strong

The tensor and ML library layer is surprisingly capable. **Candle** (Hugging Face, ~19.4k stars) provides CPU/CUDA/Metal backends with a PyTorch-like API and extensive model zoo (LLaMA, Whisper, Stable Diffusion, BERT). It shows **47% faster BERT inference** than PyTorch integration. **Burn** (~11.2k stars) is the most ambitious pure-Rust framework, with CubeCL technology enabling unified GPU/CPU kernels that compile to CUDA, Metal, or WebGPU—achieving **97% of PyTorch+CUDA performance** on Phi-3 (3.8B parameters) with **30% lower memory overhead**. **tch-rs** (~4.5k stars) provides thin, mature bindings to libtorch with full GPU support, serving as the backbone for most existing Rust RL projects.

The RL algorithm layer remains thin. **border** (LaboroAI) is the most complete Rust RL library, implementing DQN, IQN, SAC, and DDPG with both tch-rs and candle backends, async training with shared replay buffers, and OpenAI Gym wrappers via PyO3. **rurel** provides tabular Q-learning with a clean trait-based API. A newer **rl** crate built on burn aims to become the "Stable-Baselines3 of Rust" but is very early stage. None of these approaches the algorithm coverage or stability of Python alternatives.

The environment layer is where Rust already shines. **Bevy** (37k+ stars) with **Rapier** physics provides a high-performance, ECS-based game engine suitable for RL environments. **entity-gym-rs** delivers an innovative entity-based RL abstraction achieving **>100k frames/second** on an RTX 2080 Ti, with Bevy integration and browser-playable WASM demos. MuJoCo bindings exist via **mujoco-rs** (safe Rust bindings with lifetime guarantees, supporting MuJoCo 3.3.7) and **rusty_mujoco** (compile-time-safe ObjectId types). **gym-rs** provides pure Rust implementations of CartPole and MountainCar with no Python dependency.

The interop layer is production-grade. **PyO3** (13k+ stars, v0.28.2) with **maturin** is the industry standard for building Python extensions in Rust, used by Hugging Face tokenizers, OpenAI's tiktoken, Polars, Pydantic v2, and Ruff. **rust-numpy** (30M+ downloads) enables zero-copy NumPy array sharing between Rust and Python.

## Reimplementing bindings in Rust delivers quantifiable gains

The performance case rests on hard data from analogous projects. Polars outperforms Pandas by **3–10× on large datasets** (up to 22× for multi-column aggregation) with **30–60% less memory**. Pydantic v2's Rust core delivers **5–50× faster validation** (commonly 17×). Ruff lints Python **10–100× faster** than flake8. Discord's Go-to-Rust rewrite yielded **10× performance improvement**. Cloudflare's Pingora (Nginx replacement in Rust) cut infrastructure costs by **70%**. These are not micro-benchmarks—they are production systems processing real workloads.

For RL specifically, the highest-impact Rust reimplementation targets are:

- **Environment simulation**: Rust environment stepping eliminates GIL contention entirely. Rayon's data parallelism provides true parallel environment execution with a one-line change (`iter()` → `par_iter()`), delivering **4–10× speedups** over Python multiprocessing with compile-time data-race prevention. Processing 12 million log entries, Rayon cut time from 40 minutes to under 4 minutes.

- **Replay buffers**: Rust's ownership model makes buffer overflows, use-after-free, and race conditions impossible at compile time. Thread-safe shared replay buffers require zero runtime locks when designed with Rust's `Arc<Mutex<T>>` or lock-free data structures.

- **Training loop hot paths**: Rust eliminates the 2.7–6.8× Python interpreter overhead that LeanRL's `torch.compile` partially addresses. A Rust-native training loop calling into PyTorch via FFI for gradient computation gives the best of both worlds.

- **PyO3 interop**: Function call overhead is **~25–68ns**—negligible when batching operations. The pattern is proven: implement the compute-intensive inner loop in Rust, expose a Python API via PyO3, package with maturin as a pip-installable wheel. For RL, this means `env.step()` calls Rust-native simulation, `buffer.sample()` returns zero-copy tensors via rust-numpy, and the Python layer handles PyTorch model updates.

- **WebAssembly deployment**: Rust compiles to WASM that runs **8–10× faster** than equivalent JavaScript for compute-heavy tasks. Burn already supports WASM backends. This enables browser-based RL demos, edge deployment, and portable inference—impossible with Python-only frameworks.

- **Type safety**: Rust's type system catches observation/action space mismatches at compile time. rurel's trait-based API requires `State` types to declare their action types, reward functions, and transition logic—eliminating an entire class of runtime errors that plague dynamically-typed Python RL code.

## The risks are real but manageable with the right strategy

**Ecosystem immaturity** is the primary risk. Less than **5% of ML practitioners** use Rust for production AI workloads. No Rust RL library matches SB3's algorithm coverage or RLlib's distributed training. Pre-trained model availability is limited. The autograd implementations in burn and candle, while functional, lack PyTorch's battle-tested maturity across thousands of research papers.

**Developer adoption** creates a talent bottleneck. Rust's ownership and lifetime concepts are unfamiliar to ML researchers. Pydantic's creator reported that the Rust rewrite took "eight months into this three-month project." Rust developer salaries run **$150K–$210K** versus Python's $130K–$180K, with a smaller talent pool.

**GPU/CUDA integration** is improving but incomplete. CubeCL (used by burn) supports CUDA and ROCm, but there are no Rust-native cuDNN or NCCL wrappers—only FFI to C libraries. The Rust CUDA project was rebooted after being dormant for 3+ years and still carries "early development" warnings. GPU memory management in Rust requires understanding both ownership semantics and CUDA memory hierarchies.

**Development velocity** is the steepest trade-off for research workflows. RL research demands rapid iteration on reward functions, architectures, and hyperparameters. Rust's compilation step (0.8–3s incremental for Bevy, minutes for clean builds) adds friction that Python's instant execution avoids. Debug builds in Rust have poor runtime performance, requiring careful build profile configuration.

**FFI overhead** matters at the margins. PyO3's ~25–68ns per-call overhead is negligible for batched operations but can exceed computation time for trivially small work. The key principle: never call Rust from Python in a tight loop; instead, push the entire loop into Rust and return batch results.

## Conclusion: selective Rust reimplementation, not wholesale rewrite

The evidence points decisively toward the **hybrid pattern** that has succeeded across the Python ecosystem. The optimal architecture maintains Python as the researcher-facing API layer—preserving access to PyTorch's autograd, model zoo, and massive community—while reimplementing three specific bottlenecks in Rust: environment simulation and parallel stepping (eliminating the GIL), replay buffer management (gaining memory safety and throughput), and training loop orchestration (removing interpreter overhead). This delivers an estimated **5–20× throughput improvement** on environment-bound workloads based on EnvPool and Rayon benchmarks, with compile-time guarantees against the memory safety bugs that plague long-running Python RL training.

The Rust RL ecosystem's critical gap is not individual components—burn, candle, PyO3, Rapier, and Bevy are each production-capable—but integration. No project yet combines these pieces into a coherent, well-documented framework that a typical RL researcher could adopt without significant Rust expertise. The first team to build this integration layer, following the Polars model of "Rust engine, Python interface," will capture a substantial and growing market as RL workloads scale beyond what Python's architecture can efficiently support.
